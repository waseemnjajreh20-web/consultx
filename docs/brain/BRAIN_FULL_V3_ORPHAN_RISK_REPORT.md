## 3. Policy violation audit

The R3 policy gate (commit `686d4e8`) was designed to skip any `.md` whose sibling `.meta.json` had `requires_review: true`. Twelve specific SBC-201 sections were excluded by that gate:

```
sbc-201-section-102, 104, 109, 110, 111, 112, 113, 114, 115, 116, 202, 309
```

**Cross-check against v3**: 10 of these 12 sections ARE present in v3:

```
sbc-201-section-102, 109, 110, 111, 112, 113, 114, 115, 116, 309
```

(Sections 104 and 202 are NOT in v3 — possibly because round-2 added them after the v3 build, or because they were filtered out for unrelated reasons.)

### Important sub-finding

The v3 chunks themselves **do not carry the `requires_review: true` flag**. Search for `"requires_review":true` over the entire v3 chunk content returned **0 matches**. The build script that produced v3 stripped the flag from per-chunk frontmatter before emission.

This means: **if any future code ever wires v3 into the runtime, it would silently include the policy-blocked content** — the runtime has no way to detect it after the fact. The policy gate exists in the build script, not in the runtime.

This is the central risk of the v3 orphan: it is content that the project's stated policy explicitly forbids, but it sits in production and is invisible to runtime safety checks.

## 8. Disposition options

### Option A — Leave temporarily (do nothing now)
- **Risk**: latent foot-gun remains.
- **Effort**: zero.
- **Reversibility**: trivial (already there).

### Option B — Move to an `_archive/` prefix in the same bucket
- **Risk reduction**: substantial — anyone looking at `brain_full_v3` would see only an empty folder, and the archived copy lives at a path that doesn't match any plausible runtime regex.
- **Effort**: medium — requires service_role + 6 PUT + 6 DELETE operations.
- **Reversibility**: easy (move back).
- **Downside**: still in production storage; if the archived path is ever discovered and re-wired, same problem.

### Option C — Delete after backup
- **Risk reduction**: complete — the orphan is gone.
- **Effort**: medium — requires service_role + local backup + 6 DELETE operations.
- **Reversibility**: harder (need to restore from local backup).
- **Downside**: an irreversible production-state change. Should be done only after explicit operator opt-in.

### Option D — Rebuild v3 with the R3 policy gate enforced
- **Risk reduction**: turns the orphan into a policy-clean alternative to v1.
- **Effort**: high — requires running `scripts/build-consultx-brain-full.cjs` with the post-R3 gate, regenerating chunks, uploading.
- **Use case**: only valuable if there is a plan to ever read v3 from the runtime. Currently no such plan; this option is over-investment.

---

## 9. Recommendation (no execution this round)

**Recommend Option C (delete after backup).**

Reasoning:
- The orphan has no current use and no documented plan to be used.
- The policy violation in v3 is real, even though no runtime sees it today.
- Option B (archive) only moves the problem rather than removing it; same content under a different path is still available for accidental wiring.
- Option D (rebuild) adds work that delivers no benefit if the orphan is just going to be deleted anyway.
- A backup copy can be kept on the operator's local machine for a sensible retention period (e.g. 30 days) in case someone discovers a use for the v3 content; after that, it's safe to discard.

### Pre-conditions for execution (deferred to a future operator session)

1. Operator-side service_role credential (the same one R5 used; rotation deferred per owner directive).
2. Local backup of all 6 v3 files copied to `.tmp_bucket_backup/brain_full_v3_<timestamp>/`.
3. SHA256 hashes recorded for each backup file.
4. Confirmation that no PR introduces a `brain_full_v3` reference between the backup and the deletion (re-grep before delete).
5. Documented rollback: re-upload the 6 backup files to `ssss/brain_full_v3/` with the same service_role.

Total effort estimate: 30 minutes including verification. **Bucket-only operation, no code change, no deploy, no DB write.**

This recommendation is logged here. Per the brief, no deletion or move is performed in this round. The orphan stays in place.
