## 1. Current production behavior (post-revert baseline)

The Advisory pipeline (`mode === "standard"`) currently runs:

1. **Non-code intent gate** (commit `3be8214`)
   - `classifyAdvisoryIntent(userText)` returns `code_domain` | `casual` | `empty_or_ambiguous`
   - `casual` and `empty_or_ambiguous` SHORT-CIRCUIT before retrieval — emit a canned SSE reply with empty `X-SBC-Sources` / `X-SBC-Source-Meta`. No keyword retrieval, no Brain V1 sidecar download, no Evidence Ledger, no Gemini call.
   - Domain match wins over casual ("كيفك مع المشروع" routes to `code_domain`).
   - File: [supabase/functions/fire-safety-chat/index.ts:4775](supabase/functions/fire-safety-chat/index.ts:4775) (`classifyAdvisoryIntent`) and [supabase/functions/fire-safety-chat/index.ts:5344](supabase/functions/fire-safety-chat/index.ts:5344) (gate site).

2. **Structured table path** — DB-first lookup of `sbc_code_tables` for explicit table-id queries.

3. **Storage / keyword retrieval** — `fetchSBCContext` downloads chunk JSONs from `ssss/` bucket, scores via `scoreChunk` (structural-only — same as before the reverted gate), keeps `score > 0`, sorts, and selects up to `MAX_TOTAL_CONTEXT`.

4. **Empty retrieval handling** — when both structured tables and storage return nothing, the function appends a `RETRIEVAL NOTE` to the system prompt instructing the model to apply the diagnostic protocol (greet briefly, ask 1–3 clarifying questions, never cite from general memory). It does NOT return 503 (commit `4922cb3`).

5. **Evidence Ledger + Phase 2 Brain Full V1 sidecar** — runs after prompt assembly. The V1 sidecar is the Phase 3A clean version (commit `9ad06a5`), narrow-trigger regex, RELEVANT-set filter, fetched from `ssss/brain_full_v1/`.

6. **Citation Verifier** — server-side post-stream check (Step 4).

**Net effect**: greetings produce no sources; in-domain queries run the full retrieval; empty retrieval delegates to the in-prompt diagnostic protocol rather than blocking with a 503.

## 2. What was reverted, and the leading hypothesis for failure

### 2.1 Phase 3B Brain Full V2 root canary + v2-first sidecar loader
**Commit added:** `6ec141b` (Fri May 1 22:39 +0300)
**Commit reverted:** `395c63d` (Sat May 2 01:27 +0300) — ~2.5 hours later

What it did (per commit message and diff):
- Uploaded a `brain_full_v2/` corpus to bucket `ssss/`: SBC201 v2 chunks (3.2 MB), SBC801 v2 chunks (8.5 MB), `relations_v2.json`, `facts_v2.json`, `decision_tree_v2.json`, `brain_manifest_v2.json`.
- Patched `brainV1Fetch` to try `brain_full_v2/` FIRST with a v1→v2 sidecar filename alias (`relations_v1.json` → `relations_v2.json`, etc.), falling back to `brain_full_v1/` only on null result.
- Cache key changed from `key` to `${prefix}/${key}` so v1 and v2 fetches cache separately.
- "Root canary": new v2 chunk filenames (`SBC201_BrainFull_v2_canonical_chunks.json` etc.) were also placed in the bucket so `fetchSBCContext` would scan them. They had no page-range pattern in the filename, so they got default score `1` (lowest priority), only winning when keyword scoring of existing chunks was weak.

**Leading failure hypotheses (no production telemetry was preserved, so each is unverified):**

| # | Hypothesis | Reasoning |
|---|------------|-----------|
| F1 | **No fallback-on-empty.** `brainV1Fetch` falls back to v1 only when v2 returns `null` (storage error or missing object). If v2 returned a valid JSON object whose `chunks` / `relations` / `facts` arrays were empty or had a different schema than v1's RELEVANT-set filter expected, the downstream `loadBrainFullV1Sidecars` would treat it as "loaded but empty" and never fall back to v1. The prompt would lose the Phase 2 enrichment silently. | Diff shows `if (v2Hit !== null) return v2Hit` — no shape check. |
| F2 | **Schema drift between v1 and v2.** v1's RELEVANT set keys on `c.section_ref` and `f.source_refs[].replace(/^sbc-\d+-section-/, "")`. If v2 chunks/facts emitted a different field name (`section`, `sbc_ref`) or a different prefix, every chunk would be filtered out. This is exactly the silent-empty case in F1. | No upstream contract test was committed. The commit description references "round-4 local corpus regenerated by coordinator, not committed" — schema may have changed. |
| F3 | **Root canary chunks polluted retrieval as low-quality fallback.** The v2 root chunks have no page-range pattern, so they always score 1. When real keyword scoring on v1 chunks was weak (e.g. broad in-domain query without an exact section number), the multi-megabyte v2 chunks would dominate the selected set after sort, displacing tighter v1 page-range chunks and eating the `MAX_TOTAL_CONTEXT` budget with low-relevance text. | Commit description acknowledges this exact path: "v2 root chunks become the retrieval default" when v1 scoring is weak. |
| F4 | **Cache key change masked test signal.** Pre-Phase-3B, cache key was just `key`. Post-Phase-3B it is `${prefix}/${key}`. Any prior cached v1 entry was orphaned but still in memory until TTL — for the first 5 min of warm function instances, behavior depended on which cache lane the request hit. Hard to reproduce, easy to misdiagnose. | Diff at line 1199-1207 shows the cache key change. |
| F5 | **No diagnostic log of which version was used.** No `[BrainFullV2|V1] ...` log line was added to tell which file actually loaded per request. So when something looked off in production, there was no way to tell whether v2 was hit, v1 was hit, or fallback happened. | Diff has no version-tagged log. |

The revert was clean (4 lines kept of the 25 added) and reverted both the loader change and the cache-key change. The bucket uploads were not deleted — those v2 files may still be sitting in `ssss/brain_full_v2/` in the production bucket.

### 2.2 scoreChunk emission gate (relevance vs bonus split)
**Commit added:** `5db4cf0` (Sat May 2 05:18 +0300)
**Commit reverted:** `4e4f032` (Sat May 2 05:30 +0300) — 12 minutes later

What it did:
- Split `scoreChunk` scoring into two channels:
  - **relevance** — keyword + queryMeta hits (`sectionNumbers`, `tableRefs`, `exactPhrases`, `occupancyGroups`, `constructionTypes`).
  - **bonus** — structural markers (table mention, section-number presence, chapter heading).
- Added a HARD GATE: if `relevance === 0`, return `0` (chunk dropped, no chip emitted).

**Stated motivation** (from the commit message): "every SBC chunk contains those markers, every chunk passed the `score > 0` filter even when the user typed a greeting … two fallback chips appeared in the source panel on every Advisory turn".

**Leading failure hypotheses for the rapid revert (12-minute window):**

| # | Hypothesis | Reasoning |
|---|------------|-----------|
| G1 | **Arabic-query / English-chunk mismatch.** The `keywords` array passed to `scoreChunk` is built upstream from the user query plus the query-language family. SBC chunks are a mix of English and Arabic but the canonical chunks are largely English. An Arabic-only domain query like "ما متطلبات الإشغال لمحل تجاري 1200 متر مربع" may produce zero direct keyword overlap with an English chunk that genuinely contains the answer. Pre-gate behavior gave the chunk a structural bonus that kept it in play; post-gate, `relevance === 0` → drop. | Real-world Advisory queries in the AR locale are common; keyword extraction in lines 1709–1762 may or may not normalize across languages. |
| G2 | **Section-number queries without QueryMeta hit.** A user typing "متى يتطلب نظام إنذار حريق؟" has no explicit section number, no table ref, no occupancy group, no construction type, and no exact phrase a-priori. So `queryMeta` contributes 0 to relevance. If the keyword extractor produced only Arabic keywords ("إنذار", "حريق"), and the chunk text is English ("fire alarm", "Section 907"), the chunk gets `relevance === 0` → dropped. | The gate runs on EACH chunk individually; there is no global "no chunks survived → relax gate" fallback. |
| G3 | **Source panel went empty on real queries.** The reported pre-gate symptom was "two fallback chips on every greeting". Post-gate, the symptom likely flipped to "no source panel even on real queries" for AR queries. Worse failure mode: instead of misleading-but-present sources, the user sees no sources for legitimate questions. | The non-code intent gate (commit `3be8214`, added 4 days later) is the correct architectural fix for the original symptom — short-circuit BEFORE scoring runs on greetings. With that gate in place the scoreChunk gate is redundant for the casual case. |
| G4 | **No fixtures to validate behavior across languages.** No test or fixture was committed alongside `5db4cf0`. The 12-min revert window suggests the regression was caught by manual smoke or by a single live query. | No test file in `evals/` references `scoreChunk`. |

## 3. Where the failure is NOT

These were considered and ruled out as the root cause:

- **Citation verifier** — unchanged across the reverted commits; verifier failures would manifest as added "fallback to general statement" lines, not as the symptoms above.
- **Latency** — both reverted commits could only ADD latency (extra v2 fetch attempt, extra scoring). Latency alone does not explain a revert; it explains a slow but-still-correct response.
- **Routing between SBC201/SBC801** — the existing root scanner / family-routing is unchanged across both reverted commits. The Step 3.2 hard-stop wrong-family routing is intact (commit `8fdfc6f`).
- **Missing/weak corpus chunks** — v1 sidecar at `ssss/brain_full_v1/` is the production-stable corpus and is still being read. Phase 3B did NOT delete v1.

## 4. What must NOT be reintroduced blindly

1. **A v2-first sidecar loader without all of: per-key version log, fallback-on-empty (not just fallback-on-null), schema sanity check, and a kill-switch env flag.** Re-shipping `6ec141b` as-is will likely re-trigger the same silent-degradation path.
2. **A hard zero-relevance gate on `scoreChunk`.** The non-code intent gate is now in place upstream and is a more targeted fix for the original "fallback chips on greetings" symptom. A second, lower-level relevance gate at chunk-scoring time risks dropping legitimate AR-query chunks.
3. **Any change that makes empty retrieval return 503.** The diagnostic protocol (`RETRIEVAL NOTE` + system prompt) is the agreed approach — already deployed.
4. **Phase 3B v2 corpus deletion from the bucket.** The v2 files are still uploaded at `ssss/brain_full_v2/`; they are not actively read but also not actively harmful. Deletion is a separate, deliberate decision and not part of stabilization.
